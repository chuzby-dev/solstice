# Backtest tuning report — 2026-07-19

Data: real SOL/USD history from Birdeye. Coarse strategies (dca, momentum, mean-reversion,
grid, rsi-macd, volatility-breakout) backtested on 4,320 hourly candles (180 days).
Scalpers (short-window-grid, range-scalper, confluence-scalper, fee-aware-scalper)
backtested on 20,160 one-minute candles (14 days). `whale-copy` is out of scope — no
historical-replay source for on-chain transfer data exists in this codebase.

"Tuning" = first ~70% of the series (chronological), searched by random search (150
trials + the shipped defaults as a baseline). "Validation" = the held-out final ~30%,
never touched during the search — see docs/ARCHITECTURE.md "Backtesting" for full
methodology.

## The central finding: the live engine's lookback ceiling

The first full sweep (raw numbers in the appendix below) found large-looking tuning-window
improvements for `momentum`, `mean-reversion`, `rsi-macd`, and `volatility-breakout` by
stretching their period params (`lookbackPeriods`, `maPeriod`, `rsiPeriod`, etc.) — but
those periods were tuned against **1-hour candles**. Applied literally, a period of, say,
`macdSlow: 43` on 1-hour candles is a 43-*hour* lookback. Converted to the live engine's
2-second poll cadence, that's ~77,400 ticks of history. The live engine's
`ctx.priceHistory` — even after raising it in this pass (see below) — tops out at 600
ticks (~20 minutes). **A backtest-tuned period longer than that can never be satisfied
live**: `indicators.ts`'s `sma`/`ema`/`rsi`/`macd` all return `null` until they have
`period` values, so the indicator simply never computes and the strategy never fires,
regardless of how well the same numbers performed in the hourly-candle backtest.

This is a real, structural mismatch between this app's high-frequency (2s) polling design
and the multi-hour timeframes where classic TA indicators (RSI/MACD/moving averages)
actually carry a real edge — not a bug in one strategy, but a property of every
tick-count-period strategy (see `TICK_COUNT_PARAMS` in
`src/backtest/sweep.ts`). Two follow-ups came out of it:

1. **Fixed**: `strategy-engine/engine.ts`'s `ctx.priceHistory` was hardcoded to the last
   200 ticks even though `priceCache` already retains up to 600 — raised to 600 (~20min
   at the default poll rate) since that data was already being kept anyway. Purely
   additive: more available history can only let an indicator compute where it previously
   returned `null`, never forces a different signal.
2. **Not fixed (future work)**: 600 ticks still isn't enough to reach the multi-hour
   periods the coarse-candle backtest wanted. Supporting those would need either a much
   larger `priceCache` history budget (cheap — it's just small tick objects) or a slower
   secondary poll tier for swing-style strategies. Out of scope for this pass; flagged
   here rather than guessed at.

Because of this, a second, targeted re-sweep (`--hold-periods`) was run for every
tick-count-period strategy: search only the *live-safe* params (position size,
thresholds, take-profit/stop, etc.), holding the period params fixed at their shipped
values. Only a result that held up on **both** the tuning and the held-out validation
window, using only live-safe params, was applied to `registry.ts`.

## Verdict per strategy

| Strategy | Verdict | Why |
|---|---|---|
| `rsi-macd` | **Adopted** | `overboughtRsi` 70→66, `positionSizeUsd` 200→110. Tuning return -0.54%→-0.19%, validation -0.55%→-0.33% — improved on both windows, live-safe params only. Still net negative on this data, but a real, validated reduction in how bad it is. |
| `dca` | Not adopted | Tuned version's validation return (-2.87%) was *worse* than the shipped defaults' (-2.10%) — the tuning-window gain didn't generalize. |
| `momentum` | Not adopted | Full sweep's validation looked good (+0.23%) but depended on a period change that can't transfer live (see above). The hold-periods re-sweep's validation return flipped negative (+0.13%→-0.04%) — didn't hold up on live-safe params alone. |
| `mean-reversion` | Not adopted | Same pattern as momentum: hold-periods validation went from +0.11% to -0.08%. |
| `volatility-breakout` | Not adopted | Hold-periods tuning return looked great (+0.41%) but validation got markedly worse (-0.08%→-0.54%) — a textbook overfit signature. |
| `grid` | Not adopted | Shipped `lowerPrice`/`upperPrice` are intentionally `0` (unset) — the UI has the user set these from the live price sparkline. Baking in a range fitted to one historical window (`$90–$144` here) would misrepresent a config that's supposed to track *current* price, and it still lost money (-0.12% tuning) even so. |
| `short-window-grid` | Not adopted | Tuning improved (-0.09%→+0.02%) but the tuned config produced **zero** validation trades — no out-of-sample evidence either way, not a confirmed win. |
| `range-scalper` | Not adopted | Both shipped and tuned versions are flat-to-negative on this data (5-6 trades total); not enough signal to call a winner. |
| `confluence-scalper` | Not adopted | The hold-periods tuned config "improved" mostly by trading far less (69→5 tuning trades, 29→1 validation trade) — not enough validation trades to trust the result. |
| `fee-aware-scalper` | Not adopted | Returns round to ~0.00% both before and after at this strategy's small ($20-30) position size — no measurable signal in either direction on this data. |
| `whale-copy` | Out of scope | Depends on live on-chain transfer data; no historical-replay source exists. |

## How to reproduce

```bash
cd packages/backend
npm run backtest -- --strategy all --trials 150            # full sweep
npm run backtest -- --strategy rsi-macd --trials 150 --hold-periods   # live-safe-only re-sweep
```

Historical candles are cached under `packages/backend/data/backtest-cache/` — delete that
directory to force a re-fetch from Birdeye.
